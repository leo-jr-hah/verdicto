# Verdicto - Project Context

## Product
Verdicto is an **autonomous RWA asset assessment and valuation platform** with an **on-chain Verdict Oracle**.
- User submits an asset → AI agents independently value it → jurors evaluate credibility → verdict stored on-chain as a composable primitive
- **Core narrative**: "The GPS for Asset Prices" - any Casper dApp can query the Oracle via cross-contract call
- Assessment fee: 2.5 CSPR via x402 micropayment
- Oracle query fee: 0.1 CSPR per query (agent-to-agent)

## Architecture
- **dashboard/** - React + TypeScript + Vite frontend (React 19, Vite)
- **agents/** - Backend: orchestrator + 2 valuation agents + 3 jurors + trust framework + x402 micropayments
- **contracts/** - Rust smart contracts (VotingContract, ReputationRegistry, VerdictOracle) via Odra framework
- Deployed on: GCP via Cloud Run + Firebase hosting

## What Verdicto Does (the actual product flow)
1. **User submits asset** - real estate, fine art, or commodity + their asking price
2. **x402 payment** - 2.5 CSPR assessment fee collected via micropayment
3. **Dual independent valuation** - two AI agents value the same asset using different methodologies:
   - **Comps Specialist (Agent A)** - comparable sales, market transactions, price-per-sqft
   - **DCF Specialist (Agent B)** - discounted cash flow, intrinsic value, fundamentals
4. **Juror deliberation** - three specialized jurors evaluate which valuation is more credible:
   - **Evidence Analyst** - data quality, proof strength, source reliability
   - **Market Data Interpreter** - market trends, macro context, pricing signals
   - **Precedent Researcher** - historical assessments, comparable outcomes
5. **On-chain certificate** - final assessment + all reasoning committed to Casper blockchain via HMAC receipt chain
6. **Result**: a verifiable, tamper-proof assessment certificate showing the asset's AI-determined fair value

## Verdicto Points (Competition Feedback - Implemented)
1. **LTV ↔ Trust Scores** - `calculateLTV()` uses continuous formula: `ltv = base + (max - base) × confidence × valueRatio`. Low confidence → visibly lower LTV. Trust breakdown exposed in API + frontend.
2. **Escrow On-Chain** - Escrow lock (platform → borrower) and release (borrower → platform) produce real on-chain tx hashes. Visible in loan card. **Note: collateral asset is tracked off-chain (database record). Only the CSPR loan amount is on-chain. This is explicitly scoped as MVP - no on-chain collateral token lock.**
3. **Revaluation = Juror Deliberation** - Revalue re-runs dual-agent deliberation, produces HMAC receipt chain, stores commitment on-chain. Full reasoning exposed. Margin call = deliberated AI judgment.
4. **x402 on Money Flow** - Protocol-correct HTTP 402 on all money-flow endpoints: `/api/loans/create` (5 CSPR), `/api/loans/:id/repay` (2.5 CSPR), `/api/insurance/create` (3 CSPR), `/api/insurance/:id/claim` (3 CSPR). Returns `x402Version: "2"`, `paymentRequirements` with `scheme`, `supportedChains`, `payTo`, `maxAmountRequired`. Verified live with curl.
5. **Sequencing** - Escrow lock before revaluation logic.
6. **Auto-Revaluation Monitor** - `setInterval` checks active loans every 5 min. If >30 min since last revaluation, triggers dual-agent deliberation pipeline automatically. System monitors its own loans.
7. **Demo Mode** - `DEMO_MODE=true` env flag: checks every 30s with 0 staleness threshold (fires on every check). For live demos showing autonomous behavior.
8. **Admin Endpoints** - `POST /api/admin/force-revalue/:loanId` (manual revaluation trigger), `GET /api/admin/loans` (list all loans with revaluation metadata). Gated by `x-admin-secret` header matching `ADMIN_SECRET` env var (403 if missing/wrong). For demo/testing only.

## Wallet Integration (CRITICAL)
- **Official Casper Wallet SDK** (injected by extension) - NO external npm packages needed
- `window.CasperWalletProvider` is the factory function
- `window.CasperWalletEventTypes` has event names: Connected, Disconnected, ActiveKeyChanged
- Flow: `const provider = CasperWalletProvider()` → `provider.requestConnection()` → listen for events → `provider.getActivePublicKey()`
- **CRITICAL: provider.on() does NOT exist!** Events are DOM CustomEvents on `window`
- **Payment flow:** `signPayment()` in CSPRClickContext creates a native transfer deploy via `casper-js-sdk`
- Chrome Web Store: `https://chromewebstore.google.com/detail/casper-wallet/abkahkcbhngaebpcgfmhkoioedceoigp?hl=en`
- Testnet Faucet: `https://testnet.cspr.live/tools/faucet`

## Signature Format (CRITICAL — fixed 2026-06-24)
- Casper Wallet returns **raw 64-byte secp256k1 signature** (NO algorithm prefix)
- Casper RPC expects **65-byte signature** with prefix: `02` = secp256k1, `01` = Ed25519
- **MUST prepend key-type tag** before calling `Deploy.setSignature()`:
  - If `sigBytes.length === 64` → prepend `0x02` (secp256k1) → 65 bytes
  - If `sigBytes.length === 65` and first byte is `0x01` or `0x02` → already tagged, pass through
- Without the tag, RPC returns: `"Expected 0, 1 or 2, got 67"` (0x43 = raw sig data)
- Fix is in `CSPRClickContext.tsx` → `signPayment()` function

## Dispute & Re-trial System (NEW - June 2026)
- **What it is**: Challenge any oracle verdict with 5 CSPR stake → adversarial AI panel re-evaluates
- **Backend models**: `Dispute`, `RetrialResult`, `RetrialJuror` in `casper-contracts.ts`
- **Backend store**: `disputeStore` in-memory, seeded with 1 resolved/overturned + 1 pending dispute
- **Endpoints**:
  - `POST /api/oracle/dispute` — file dispute (x402 gated, 5 CSPR)
  - `POST /api/oracle/disputes/:id/retrial` — trigger adversarial re-trial
  - `GET /api/oracle/disputes` — list all disputes
  - `GET /api/oracle/disputes/:id` — get dispute detail
- **Adversarial panel** (3 independent agents, different from original jurors):
  - Adversarial Market Analyst (stress_test) — worst-case scenarios
  - Deep Value Auditor (value_at_risk) — downside risk evaluation
  - Devil's Advocate (contrarian) — argues against consensus
- **Stake resolution**: overturned → challenger gets stake back; upheld → stake to original jurors
- **Overturned verdicts** update the oracle store with corrected value
- **Frontend**: Full dispute section with status badges, outcome badges, expanded detail showing original vs new verdict, per-juror methodology + vote + reasoning, stake distribution
